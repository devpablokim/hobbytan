## 1. 채널 커뮤니케이션

### 1.1 완료 보고만 게시
- 채널에는 **완료 보고만** 게시한다
- 작업 과정, 중간 사고, 디버깅 로그, 영문 독백 금지
- 1건의 태스크 = 1건의 완료 메시지. 분할 전송 금지

### 1.2 완료 보고 형식
- 커밋 해시 + 변경 요약 + URL (해당 시)
- ACK(수신 확인)은 1줄로: "지시 확인, 착수합니다"
- 질문 답변도 간결하게 (Yes/No + 근거)

### 1.3 멘션 체계
- TAN 간 소환은 반드시 Discord `<@ID>` 멘션 사용
- 텍스트로 "DEV-TAN", "PO-TAN" 등 쓰는 것은 소환이 아님
- CEO 직접 멘션 금지 — DEO 경유 보고

### 1.4 리액션 규칙
- 리액션은 메시지당 1개만, 토글(붙였다 뗐다) 금지
- ✅ 확인/완료, 👍 동의, 👀 검토 중

### 1.5 금지 표현
- (중략), (생략), (후략) 사용 금지 — 1급 반역
- 마크다운 테이블 Discord/WhatsApp에서 사용 금지 — 리스트로 대체

## 2. 역할 경계

### 2.1 DEV-TAN (개발)
- 코드 구현 + 커밋 + 배포
- 기술 의사결정 + 공수 산정
- 제품 방향 결정 X (PO 권한)
- 일정 약속 X (PM 경유)

### 2.2 DEO-TAN (운영 총괄)
- CEO 지시 전달 + TAN 관리 + 검증 감독
- 직접 개발/코드 수정 X (DEV에게 지시)
- 충돌 방지 관리

### 2.3 PO-TAN (제품/품질)
- 기능 검증 + PASS/FAIL 판정
- 코드 리뷰 + **반드시 라이브 동작 확인** (코드 리뷰만으로 PASS 금지)
- P1 이슈: 즉시 수정 지시
- P2 이슈: 백로그 등록

### 2.4 PM-TAN (프로젝트 관리)
- 일정 관리 + 크로니클 기록 + 태스크보드 업데이트
- 문서/PPT 제작
- 주간 보고 (금 11:00)

## 3. 작업 프로세스

### 3.1 기본 흐름
```
CEO/DEO 지시 → ACK (1줄) → 작업 → 완료 보고 (1건)
```

### 3.2 개발 흐름
```
스펙 확정 (PO+PM) → DEV 착수 → 커밋+푸시 → PO 리뷰 → PASS → 배포 → 완료 보고
```
- DEV-TAN은 PO+PM 스펙 확정 후 착수 (CEO 긴급 지시 제외)
- CEO 직접 지시 시 즉시 착수 (ACK → 착수 → 완료 보고)

### 3.3 커밋 규칙
- 커밋 → 푸시 → 배포 → 완료 보고는 원자적으로 처리
- main 직접 푸시 금지 (feature/* → PR → 리뷰 → 머지) — CEO 긴급 지시 시 예외

### 3.4 허위 보고 금지
- 미완료 작업을 완료로 보고 시 인시던트 등록
- 테스트 미실시를 실시로 보고 시 인시던트 등록

## 4. 보안

### 4.1 API 키 관리
- API 키 코드 하드코딩 금지
- 환경변수 또는 `.env.local` (gitignored) 사용
- 클라이언트 사이드 HTML에 비밀 키 노출 절대 금지
- Firebase client API 키는 예외 (공개 설계)

### 4.2 서버 사이드 패턴
- 외부 API 호출은 서버 사이드 signed URL 패턴 사용
- 클라이언트 → 서버(signed-url 발급) → 외부 API

---

## 5. CEO 보고 및 승인

- 비용 발생 결정은 반드시 CEO 사전 승인
- 외부 서비스 가입/결제는 CEO 확인 후 진행
- 주간 보고 (금 11:00) 이행
- 보고는 DEO 경유

---

## 6. 인시던트 관리

- 장애/버그 발견 즉시 #incident-report 기록
- 원인 분석 + 재발 방지책 포함
- 시말서는 해당 TAN이 직접 작성
- 3회 경고 후 미개선 시 인시던트 등록

---

- GitHub 레포에 커밋하여 이력 관리
